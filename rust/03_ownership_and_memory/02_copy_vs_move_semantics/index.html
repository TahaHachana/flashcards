Copy vs Move Core Distinction

What's the core difference between copy and move semantics?

Copy: value is duplicated, both variables have independent copies. Move: ownership transfers, original variable becomes invalid. Only the new variable can access the value.

Why Two Different Semantics

Why does Rust have both copy and move semantics?

Performance (copying large data is expensive), safety (move prevents double-free by ensuring one owner), and ergonomics (small types like integers should be usable naturally).

The Copy Trait Requirements

What requirements must a type meet to implement Copy?

  1. All parts live on the stack (no heap allocation)
  2. No custom cleanup needed (no Drop implementation)
  3. Bit-wise copy creates a valid, independent instance

Move is the Default

Which semantics does Rust use by default: copy or move?

Move is the default. Unless a type explicitly implements Copy, it uses move semantics. This is the safe default - assume ownership transfer.

Scalar Types Are Copy

Which scalar types implement Copy?

All of them: integers (i8, i32, u64, etc.), floats (f32, f64), bool, and char. They're small and live entirely on the stack.

Tuples Copy Rule

When is a tuple Copy?

Only if all its elements are Copy. If any element uses move semantics, the entire tuple does too.

let a: (i32, i32) = (1, 2);        // Copy
let b: (i32, String) = (1, s);     // NOT Copy

Arrays Copy Rule

When is an array Copy?

Only if its element type is Copy.

let a: [i32; 3] = [1, 2, 3];              // Copy
let b: [String; 2] = [s1, s2];            // NOT Copy

Why String Isn't Copy

Why doesn't String implement Copy?

String owns heap-allocated data. If copied, both would point to the same heap. When going out of scope, both would try to free it (double-free bug). Move semantics prevent this.

Why Vec Isn't Copy

Why doesn't Vec implement Copy?

Vec manages heap-allocated buffer. Same double-free issue as String - if both owned the same buffer, both would try to free it when going out of scope.

Custom Struct Copy Rule

When can a custom struct be Copy?

Only if all fields are Copy. If any field uses move semantics, the entire struct must use move semantics.

struct Point { x: i32, y: i32 }  // Can be Copy
struct User { name: String }      // Cannot be Copy

Copy in Assignment

What happens when you assign a Copy type to a new variable?

The value is duplicated. Both variables are valid and independent.

let x = 5;
let y = x;
println!("{}, {}", x, y);  // ✅ Both valid

Move in Assignment

What happens when you assign a non-Copy type to a new variable?

Ownership moves. The original variable becomes invalid.

let s1 = String::from("hello");
let s2 = s1;
println!("{}", s1);  // ❌ Error: s1 moved

Copy in Function Calls

What happens when you pass a Copy type to a function?

The value is copied. The original remains valid after the call.

fn use_num(n: i32) { }
let num = 42;
use_num(num);
println!("{}", num);  // ✅ Still valid

Move in Function Calls

What happens when you pass a non-Copy type to a function?

Ownership moves to the function. The original becomes invalid.

fn use_str(s: String) { }
let text = String::from("hello");
use_str(text);
println!("{}", text);  // ❌ Error: text moved

What Actually Moves

When ownership moves for heap-allocated types, what gets copied?

Only the stack portion (pointer, length, capacity, etc.). The heap data isn't duplicated. The old variable is marked invalid to prevent double-free.

// Stack: [ptr, len, cap] copied
// Heap: same data, not duplicated
// s1: marked invalid

Moves Are Cheap

What is the performance cost of moving ownership?

Very cheap - just copying a few bytes on the stack (pointer, length, capacity). The heap data isn't copied. The old variable is invalidated.

Moves Are Implicit

Do you need special syntax to move ownership?

No. Moves happen automatically on assignment and function calls. The compiler tracks that moved variables are invalid.

let s1 = String::from("hello");
let s2 = s1;  // Implicit move

Making Custom Type Copy

How do you make a custom struct Copy?

Derive both Copy and Clone. The type must contain only Copy types and not implement Drop.

#[derive(Copy, Clone)]
struct Point {
    x: i32,
    y: i32,
}

Copy Requires Clone

Why must you derive both Copy and Clone?

Copy is a subtrait of Clone. All Copy types must also implement Clone, so you must derive both.

Drop Prevents Copy

Can a type implement both Copy and Drop?

No. A type with Drop cannot be Copy. Drop implies special cleanup, which doesn't make sense for a simple bit-wise copy.

Clone Explicit Copying

How do you explicitly duplicate a non-Copy type?

Use .clone() to create a deep copy. This allocates new memory and copies all data.

let s1 = String::from("hello");
let s2 = s1.clone();
println!("{} {}", s1, s2);  // ✅ Both valid

Clone vs Copy Differences

What are the key differences between Clone and Copy?

Copy: implicit (automatic), cheap (bit-wise), stack-only types. Clone: explicit (must call .clone()), can be expensive (may allocate heap), works for any type implementing Clone.

Why Clone Makes Cost Visible

Why does Rust require explicit .clone() calls?

Makes the cost visible. When you see .clone(), you know a potentially expensive operation is happening (heap allocation and data copying), helping identify optimization opportunities.

When to Use Copy for Structs

When should you make a custom struct Copy?

When all fields are small and Copy, the type represents a simple value (coordinates, colors, IDs), and implicit copying is convenient and safe.

When to Use Move for Structs

When should a custom struct use move semantics?

When any field allocates (String, Vec, Box), the type manages resources, or you want to enforce single ownership.

Passing Copy Types to Functions

Should you pass Copy types by value or reference?

By value freely. Copying is cheap and the caller keeps the original.

fn distance(p1: Point, p2: Point) -> f64 {
    // p1, p2 are copies
}

Copy Performance

Is copying always fast?

Copy is fast for small types (few bytes). It's only available for stack-only types, so it's always cheap bit-wise copying.

Move Performance

Is moving always fast?

Yes. Moving large types like Vec is fast because it only copies the stack portion (pointer/len/cap), not the heap data.

let v1 = vec![0; 1_000_000];
let v2 = v1;  // Fast: just copies pointer

Clone Performance

Is cloning always fast?

No. Clone can be slow because it allocates and copies all data, including heap contents.

let v1 = vec![0; 1_000_000];
let v2 = v1.clone();  // Slow: copies million elements

Type System Determines Semantics

Who decides whether a value is copied or moved?

The type system, based on whether the type implements Copy. You don't choose per-operation - it's determined by the type's definition.
