Two Core Borrowing Rules

What are the two core borrowing rules?

  1. At any time: either one mutable reference OR any number of immutable references
  2. References must always be valid: cannot outlive data, must always point to valid data

Why Rule 1 Exists

Why does Rule 1 (exclusive mutable or shared immutable) exist?

Prevents data races. Multiple mutable references or mixing mutable with immutable creates situations where code could interfere with itself unpredictably, causing conflicts and bugs.

Why Rule 2 Exists

Why does Rule 2 (references must be valid) exist?

Prevents use-after-free. If references could outlive their data, you'd access freed memory (undefined behavior). Ensures references always point to valid data.

What is The Borrow Checker

What is the borrow checker?

The part of the compiler that enforces borrowing rules. It tracks which variables own data, which references exist, when references are created/used, and whether they're mutable or immutable. Produces compile errors for violations.

Why Mutable References Are Exclusive

Why can you only have one mutable reference at a time?

Prevents data races and memory corruption. If multiple mutable references existed, operations like reallocation could invalidate other references, causing crashes.

let r1 = &mut vec;
let r2 = &mut vec;  // ❌ Error
// r1.push() could invalidate r2

Why Mutable and Immutable Cannot Coexist

Why can't mutable and immutable references exist simultaneously?

Immutable references expect data won't change. Mutable references can change data, violating immutable references' expectations.

let r1 = &s;      // Expects s unchanged
let r2 = &mut s;  // ❌ Could change s

Why Multiple Immutable Are Safe

Why can you have unlimited immutable references?

Reading data from multiple places simultaneously is safe. All references are read-only, so no one can modify data and no conflicts are possible.

let r1 = &s;
let r2 = &s;
let r3 = &s;  // ✅ All safe

Preventing Dangling References

How does the borrow checker prevent dangling references?

Tracks lifetimes and ensures references can't outlive the data they point to.

let r;
{
    let x = 5;
    r = &x;  // ❌ Error: x doesn't live long enough
}
// x dropped, r would be dangling

Valid Reference Lifetimes

When is a reference's lifetime valid?

When it doesn't outlive the data it references. The reference's lifetime must be within the data's lifetime.

let x = 5;
let r = &x;  // ✅ OK: r's lifetime within x's
println!("{}", r);

Cannot Return Reference to Local

Why can't you return a reference to a local variable?

Local variable is dropped when function ends, making the reference dangle. Must return owned value instead.

fn dangle() -> &String {
    let s = String::from("hello");
    &s  // ❌ Error: s dropped, reference invalid
}

What Are Non-Lexical Lifetimes

What are Non-Lexical Lifetimes (NLL)?

Modern Rust feature where references are valid from creation until last use, not the entire scope. Makes borrowing more flexible and intuitive.

let r = &s;
println!("{}", r);  // Last use
s.push_str("!");  // ✅ OK: r no longer active

NLL Practical Impact

How do Non-Lexical Lifetimes improve usability?

References end when you're done using them, not at scope boundaries. Allows operations after a reference's last use.

let first = &vec[0];
println!("{}", first);  // Last use
vec.push(4);  // ✅ OK with NLL

Borrowing Doesn't Transfer Ownership

Does borrowing transfer ownership?

No. Ownership remains with the original owner. The function just borrows temporarily.

fn use_str(s: &String) { }
let s = String::from("hello");
use_str(&s);  // Borrow
println!("{}", s);  // ✅ Still valid

Cannot Borrow Moved Value

Can you borrow a value after it's been moved?

No. The original variable no longer owns anything, so there's nothing to borrow.

let s = String::from("hello");
let s2 = s;  // Move
let r = &s;  // ❌ Error: borrowed after move

Borrowing Prevents Moving

Can you move a value while it's borrowed?

No. If move were allowed, references would become dangling.

let r = &s;
let s2 = s;  // ❌ Error: can't move while borrowed
println!("{}", r);

Sequential Mutable References

Can you have multiple mutable references if they don't overlap in time?

Yes. As long as they're not active simultaneously.

let r1 = &mut s;
println!("{}", r1);  // Last use
let r2 = &mut s;  // ✅ OK: r1 no longer active

Mutable Blocks Immutable

Can you create an immutable reference while a mutable reference exists?

No. Immutable reference expects data won't change, but mutable reference can change it.

let r1 = &mut s;
let r2 = &s;  // ❌ Error
r1.push_str("!");

Separate Read Write Phases

What's the correct pattern for mixing immutable and mutable borrows?

Clear separation: read phase with immutable references first, then write phase with mutable reference after immutable refs are done.

let r1 = &s;
println!("{}", r1);  // Read phase ends
let r2 = &mut s;  // Write phase begins

Borrowing Individual Struct Fields

Can you borrow multiple fields of a struct mutably?

Yes. Different fields are independent data, so borrowing them separately is safe.

let x_ref = &mut point.x;
let y_ref = &mut point.y;  // ✅ OK

Cannot Borrow Whole While Field Borrowed

Can you borrow an entire struct while one of its fields is already borrowed?

No. Borrowing the whole struct would provide access to the already-borrowed field.

let x_ref = &mut point.x;
let point_ref = &point;  // ❌ Error

What is Reborrowing

What is reborrowing?

Creating a new reference from an existing mutable reference. Allows temporary sharing while maintaining the original reference.

let r1 = &mut x;
let r2 = &mut *r1;  // Reborrow
*r2 = 10;
*r1 = 20;  // r1 still valid

Implicit Reborrowing

When does implicit reborrowing occur?

When passing a mutable reference to a function expecting an immutable reference.

fn read(s: &String) { }
let r = &mut s;
read(r);  // Implicit reborrow as &String
r.push_str("!");  // r still valid

Interior Mutability

What is interior mutability?

Mutation through shared references using types like RefCell. Uses runtime checking instead of compile-time. Wrong usage causes panic, not compile error.

let data = RefCell::new(5);
let r = data.borrow_mut();  // Runtime checked

When Interior Mutability Needed

When is interior mutability needed?

Rare cases: reference-counted data needing mutation, scenarios where borrow checker is too conservative, or implementing certain data structures. Trade-off: runtime checks vs compile-time guarantees.

Pattern Conditional Borrowing

How do you limit a borrow to only where it's needed?

Use scopes or conditionals to borrow only in the specific scope.

if data.len() > 2 {
    let slice = &data[0..2];
    println!("{:?}", slice);
}  // Borrow ends
data.push(4);

Pattern Early Drop

How do you end a borrow before the natural scope ends?

Use explicit scope blocks to end borrows early.

{
    let r = &data[..];
    println!("{:?}", r);
}  // Borrow ends explicitly
data.push(4);

Data Race Prevention

How does the borrow checker prevent data races?

Enforces that only one mutable reference exists at a time. Multiple mutable references trying to modify simultaneously would cause data races.

let r1 = &mut counter;
let r2 = &mut counter;  // ❌ Prevents data race

Iterator Invalidation Prevention

How does the borrow checker prevent iterator invalidation?

Can't modify a collection while iterating. Common bug in C++.

for item in &vec {
    vec.push(*item);  // ❌ Error: can't modify while iterating
}

Borrow Checker Is Conservative

Why does the borrow checker sometimes reject safe code?

It's conservative - rejects code if it can't prove it's safe. This is intentional to maintain safety guarantees. Better to reject safe code than allow unsafe code.

Compile-Time Enforcement Benefits

What are the benefits of compile-time enforcement of borrowing rules?

Zero runtime overhead (no runtime checks needed), bugs caught before running (can't compile unsafe code), and guaranteed safety (entire categories of bugs prevented: data races, use-after-free, dangling pointers, iterator invalidation).
