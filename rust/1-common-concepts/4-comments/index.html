Purpose of Comments

What is the primary purpose of comments in Rust?

Comments are used to explain code for human readers. They are ignored by the compiler and do not affect program behavior or performance.

Single-Line Comment

How do you write a single-line comment in Rust?

Use // followed by your comment text:

// This is a single-line comment

The comment continues to the end of the line.

Multi-Line Comments

How do you write multi-line comments in Rust?

Use // at the beginning of each line:

// This is a multi-line comment
// explaining something complex
// across multiple lines

Block Comments

Does Rust have block comments like /* */ in other languages?

While Rust does support /* */ block comments, the idiomatic way is to use // at the beginning of each line for multi-line comments.

Comment Placement Options

What are the two main placement options for comments in Rust?

  1. Above the code (preferred for clarity):
// I'm feeling lucky today
let lucky_number = 7;
  1. Inline at the end of a line:
let lucky_number = 7; // I'm feeling lucky today

Preferred Placement

Which comment placement is generally preferred and why?

Above the code is preferred for clarity. It makes comments more visible and easier to read, especially for longer explanations.

Documentation Comments Syntax

What syntax is used for documentation comments in Rust?

Use /// (three slashes) for documentation comments:

/// This function adds two numbers together
fn add(a: i32, b: i32) -> i32 {
    a + b
}

Comments and Program Behavior

Do comments affect the compiled program in any way?

No, comments are completely ignored by the compiler. They don't affect program behavior, performance, or binary size.

Fix Comment Syntax

Fix the comment syntax in this code:

/* This is wrong */
let x = 5;

// This is correct
let x = 5;

While /* */ works, // is the idiomatic way in Rust.

Regular vs Doc Comments

What's the difference between // and /// comments?

  • //: Regular comments for code explanation
  • ///: Documentation comments processed by rustdoc to generate API documentation

Algorithm Comment Placement

Where should you place a comment explaining a complex algorithm?

Above the code block, before the algorithm starts:

// Implement quicksort algorithm
// Time complexity: O(n log n) average case
fn quicksort(arr: &mut [i32]) {
    // implementation
}

Inline Comments Use

When might you use inline comments (at the end of a line)?

For brief explanations or clarifications:

let tax_rate = 0.08; // 8% sales tax
let pi = 3.14159;    // Approximation for calculations

Comments Inside Expressions

Can you put comments inside expressions or statements?

Yes, but it's generally not recommended as it can hurt readability:

let result = x + /* this is confusing */ y;
// Better:
// Add x and y together
let result = x + y;

Nested Comments

What happens if you try to nest comments in Rust?

Regular // comments can't be nested since they go to end of line. Block comments /* */ can be nested:

/* This /* nested */ comment works */

But // style is preferred.

Avoid Obvious Comments

Should you comment obvious code in Rust?

No. Avoid commenting obvious code. Focus on why rather than what:

// Bad
let x = 5; // Assign 5 to x

// Good
let max_retries = 5; // Limit to prevent infinite loops

Doc Comments and Tooling

How do documentation comments integrate with Rust tooling?

Documentation comments (///) are processed by rustdoc to automatically generate HTML documentation for a crate's public API.

Commenting Parameters

What's a good practice for commenting function parameters?

Use documentation comments to explain parameters:

/// Calculates the area of a rectangle
///
/// # Arguments
/// * `width` - The width in pixels
/// * `height` - The height in pixels
fn area(width: u32, height: u32) -> u32 {
    width * height
}

When to Comment

Should you comment every line of code?

No. Over-commenting hurts readability. Comment: - Complex logic - Business rules - Non-obvious decisions - Public API functions (with ///)

Temporarily Disable Code

How do you temporarily disable code using comments?

Comment out the lines:

fn debug_function() {
    println!("Debug info");
    // println!("Temporary disabled");
    // let temp = calculate_something();
}

Updating Comments

What's the best practice for updating comments?

Keep comments in sync with code changes. Outdated comments are worse than no comments because they mislead readers. Update comments whenever you modify the related code.
