What is a Merge Conflict

What is a merge conflict and when does it occur?

A merge conflict occurs when Git cannot automatically resolve differences between two branches. This happens when: - Both branches modify the same lines in the same file - One branch deletes a file while another modifies it - Both branches create files with the same name - Both branches modify binary files

Git pauses the merge and asks you to manually decide which changes to keep. Conflicts are normal in collaborative development.

Detecting Merge Conflicts

How do you detect when a merge conflict has occurred?

When attempting merge:

git merge feature-branch
# CONFLICT (content): Merge conflict in file.txt
# Automatic merge failed; fix conflicts and then commit

Check status:

git status
# You have unmerged paths.
# Unmerged paths:
#   both modified:   src/app.js

Key indicators: - CONFLICT message - Automatic merge failed - List of conflicting files in status

Understanding Conflict Markers

What do the conflict markers mean in a file with merge conflicts?

Git adds special markers to show conflicting sections:

<<<<<<< HEAD
Changes from current branch (main)
=======
Changes from branch being merged (feature)
>>>>>>> feature-branch

Three sections: 1. <<<<<<< HEAD: Start, shows current branch changes 2. =======: Separator between versions 3. >>>>>>> branch-name: End, shows incoming changes

You must choose which to keep (or combine) and remove ALL markers.

Basic Conflict Resolution Steps

What are the 8 steps to resolve a merge conflict?

  1. Identify conflicting files: git status
  2. Open each conflicting file in editor
  3. Locate conflict markers (<<<<<<<)
  4. Decide resolution strategy (keep ours, theirs, combine, or new)
  5. Remove ALL conflict markers completely
  6. Stage resolved file: git add file.txt
  7. Repeat for all conflicts
  8. Complete merge: git commit

Critical: Must remove all <<<<<<<, =======, >>>>>>> markers!

Conflict Resolution Strategies

What are four strategies for resolving a conflict between two versions?

Given conflict:

<<<<<<< HEAD
console.log("Welcome to our app!");
=======
console.log("Welcome, user!");
>>>>>>> feature
  1. Keep current (ours): "Welcome to our app!"
  2. Keep incoming (theirs): "Welcome, user!"
  3. Combine both: "Welcome to our app, user!"
  4. Write new solution: console.log(\Welcome ${username}!`);`

Choose based on understanding both changes and requirements.

Accepting Ours or Theirs

How do you quickly accept all changes from one side of a conflict?

Accept current branch (ours):

git checkout --ours file.txt
git add file.txt

Accept incoming branch (theirs):

git checkout --theirs file.txt
git add file.txt

For all files:

git checkout --ours .    # All ours
git checkout --theirs .  # All theirs
git add .

⚠️ Use carefully—resolves without reviewing individual changes.

Aborting a Merge

How do you cancel a merge with conflicts and when should you do it?

Cancel merge:

git merge --abort

This: - Cancels the merge operation - Returns to pre-merge state - Discards all conflict resolutions - Safe to use anytime before final commit

Use when: - Made mistakes in resolution - Conflicts too complex - Wrong branch merged - Need to prepare differently

Using Git Merge Tool

How do you use Git's merge tool for resolving conflicts?

Configure merge tool:

# VS Code
git config --global merge.tool vscode
git config --global mergetool.vscode.cmd 'code --wait $MERGED'

# Or vimdiff, meld, etc.
git config --global merge.tool vimdiff

Launch during conflict:

git mergetool

Opens configured tool with visual interface for resolving conflicts. Modern IDEs (VS Code, IntelliJ) have built-in conflict resolution UIs.

Complete Resolution Example

Walk through a complete conflict resolution example.

# Attempt merge
git merge feature/login
# CONFLICT in auth.js

# Check status
git status
# both modified: auth.js

# Edit file (remove markers, fix conflict)
code auth.js

# Stage resolved file
git add auth.js

# Verify all resolved
git status
# All conflicts fixed

# Complete merge
git commit -m "Merge feature/login"

# Test
npm test

Key: Edit → Remove markers → Stage → Commit → Test

Preventing Merge Conflicts

What are five best practices to prevent merge conflicts?

  1. Pull frequently: git pull origin main (stay updated)
  2. Keep feature branches short-lived (create → work → merge quickly)
  3. Communicate with team (coordinate who works on what)
  4. Break features into smaller pieces (smaller changes = fewer conflicts)
  5. Merge main into feature regularly: bash git switch feature-branch git merge main

Prevention is better than resolution. Communication and frequent integration reduce conflicts.

Testing After Resolution

Why must you test after resolving conflicts and what should you test?

Conflicts might introduce logical errors even if syntactically correct. Always test:

  1. Run test suite: bash npm test pytest

  2. Verify functionality:

  3. Manual testing of affected features

  4. Check integration points

  5. Build/compile: bash npm run build

Merged code might work individually but fail together. Testing catches integration issues.

Conflict Resolution Best Practices

What are key best practices during and after conflict resolution?

During resolution: - Read both versions carefully - Understand intent of each change - Test after resolving each file if possible - Communicate with commit authors if unclear - Preserve functionality over style

After resolution: - Run full test suite - Manual testing of affected features - Code review if working in team - Document resolution in merge commit message

Always prioritize correctness and functionality.

Key Conflict Resolution Principles

What are the essential principles to remember about resolving merge conflicts?

  1. Conflicts are normal in collaborative development
  2. Conflict markers show three sections (HEAD, separator, incoming)
  3. Always remove ALL markers completely before committing
  4. git add marks resolution for each file
  5. Test after resolving to catch logical errors
  6. git merge --abort safely cancels problematic merges
  7. Communication prevents conflicts better than resolution skills
  8. Short-lived branches reduce conflict likelihood
  9. Use tools (IDE, mergetool) for complex conflicts
  10. Document resolutions in commit messages

Practice makes perfect—conflict resolution is a critical Git skill!
