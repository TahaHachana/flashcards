What Is Version Control?

What is a version control system, and what three fundamental questions does it answer?

A version control system (VCS) records changes to files over time so specific versions can be recalled later. It answers: 1. What changed? — the exact modifications made 2. Who changed it? — the author of the change 3. When and why? — the timestamp and the commit message

Three Generations of VCS

What are the three generations of version control systems, and a key example of each?

  1. Local VCS — history stored in a database on a single machine (e.g., RCS)
  2. Centralized VCS (CVCS) — one central server holds all history; clients check out files (e.g., SVN, CVS)
  3. Distributed VCS (DVCS) — every client fully mirrors the entire repository (e.g., Git, Mercurial)

Single Point of Failure in CVCS

What is the critical weakness of a Centralized VCS (CVCS)?

A CVCS has a single point of failure: if the central server goes down, no one can collaborate or access history. If the central disk is corrupted with no backups, the entire project history is permanently lost. Every client only has the latest snapshot, not the full history.

Git Creation and Goals

Who created Git, when, and what were the original design goals?

Linus Torvalds created Git in 2005 after the Linux kernel lost access to BitKeeper. His goals were: - Speed - Simple design - Strong support for non-linear development (thousands of branches) - Fully distributed — no single authoritative server - Capable of handling large projects efficiently

Snapshots vs. Deltas

How does Git store data differently from most other VCS tools like SVN?

Most VCS tools store data as a list of file-based changes (deltas) over time.

Git stores data as snapshots — at each commit, Git takes a picture of all files and stores a reference to that snapshot. If a file hasn't changed, Git stores a link to the previous identical version rather than the file again. Git thinks of data more like a mini filesystem.

Local Operations Speed

Why is Git extremely fast for most operations compared to a CVCS?

Because Git mirrors the entire repository history locally, nearly every operation only requires local disk access — no network round-trips needed. This includes browsing history, comparing versions, and switching branches. Operations that would take seconds over a network are instantaneous locally.

Git Integrity via SHA-1

How does Git ensure the integrity of stored data?

Everything in Git is checksummed with a SHA-1 hash — a 40-character hexadecimal string — before being stored. Git refers to everything by this hash. It is impossible to change any file or directory content without Git detecting it. Example hash:

24b9da6552252987aa493b52f8696cd6d3b00373

The Three States of Git Files

What are the three states a file can be in within Git, and where does each reside?

State       | Location              | Meaning
------------|----------------------|----------------------------------------
Modified    | Working Directory    | Changed but not yet committed
Staged      | Staging Area (Index) | Marked to go into the next commit
Committed   | .git Repository      | Safely stored in the local database

Three Sections of a Git Project

What are the three main sections of a Git project?

  1. Working Directory — a single checkout of one version of the project, placed on disk for you to modify
  2. Staging Area (Index) — a file in .git that stores what will go into the next commit
  3. Git Repository (.git directory) — where Git stores the object database and metadata; what is copied when you clone

Basic Git Workflow

What are the three steps of the basic Git workflow?

  1. Modify files in your working directory
  2. Stage the changes you want to include — selectively adding them to the staging area
  3. Commit — Git takes the staged snapshot and stores it permanently in the .git repository

Git as a Safety Net

Why is it said that Git "generally only adds data"? Why is this important?

Almost every Git action adds data to the Git database rather than deleting or overwriting it. Once a snapshot is committed, it is extremely difficult to lose it. This means experimentation is safe — you can almost always undo or recover from mistakes, making Git a very forgiving system to work with.

What Is a Commit?

What is a Git commit?

A commit is a snapshot of the entire project at a specific point in time, stored permanently in the Git repository. Each commit is identified by a unique SHA-1 hash and records: - The snapshot of staged files - The author and timestamp - A commit message describing the change - A pointer to the parent commit(s)

Advantages of DVCS Over CVCS

What are the three main advantages of a Distributed VCS (like Git) over a Centralized VCS?

  1. Every clone is a full backup — the entire history is mirrored on every developer's machine
  2. Most operations are local — no network required for history, diffs, or branching
  3. Multiple workflows — supports flexible collaboration patterns like feature branches, forks, and multiple remotes without a mandatory central authority
